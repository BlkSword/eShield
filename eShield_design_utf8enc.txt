2.1.1 eBPF / XDP 内核支持矩阵

XDP（eXpress Data Path）允许在 Linux 内核网络栈的最底层——甚至在网卡驱动程序中、在

sk_buff                          结构体分配之前——执行沙盒化的                                eBPF

字节码。这意味着恶意数据包可以在进入协议栈、消耗 CPU 进行路由查找和 socket

匹配之前就被丢弃，单核丢包性能可达 5-10 Mpps（每秒百万包），比传统 Netfilter/iptables

路径快 10-100 倍。

2.1.3 CO-RE 与单二进制分发

CO-RE（Compile                  Once  -       Run             Everywhere）是  eBPF

程序跨内核版本运行的关键技术。其核心机制是：编译时将                                                 eBPF

程序中所有对内核结构体字段的访问记录为"重定位记录"（relocation

records），运行时根据目标内核的                               BTF（BPF                  Type

Format）信息动态调整字段偏移。这使得一个编译好的 .o 文件可以在 5.10 到 6.6

的内核上无缝运行，彻底解决了传统 kprobe 模块需要为每个内核版本单独编译的噩梦。

配合 Rust 的 Aya 框架，可以将 eBPF 字节码（约 100-300 KB）和用户态控制程序（约 3-5

MB）静态链接成单一 ELF 可执行文件。用户只需 curl 下载一个文件、chmod +x、./eshield

start                                                        即可运行，无需安装

clang、llvm、libbpf、内核头文件等任何依赖。这是面向普通用户分发的关键前提。

2.1.4 Ring Buffer 替代 Perf Buffer

内核 5.8 引入的 BPF_MAP_TYPE_RINGBUF 是事件传输的范式转移。相比传统的 Perf
Buffer，Ring Buffer 具有三大优势：

• 无锁单生产者多消费者：内核态写入用自旋锁保护，用户态读取完全无锁，CPU 开销降低
3-5 倍。

• 零拷贝读取：用户态可以直接 mmap Ring Buffer 内存，避免 Perf Buffer
的两次内存拷贝。

• 保证事件顺序：单 Ring Buffer 内事件严格有序，便于用户态重建攻击时序。

这使得实时监控面板（TUI/Web）可以以极低开销展示每个被丢弃的数据包的源                                     Buffer
IP、协议、命中规则，而不会反过来拖累防护性能。在 100 万 PPS 攻击下，Ring
的事件传输 CPU 开销可控制在 1% 以内。

2.1.5 Aya 框架：纯 Rust eBPF 工具链

Aya 是 2022 年后崛起的纯 Rust eBPF 框架，相比传统的 libbpf + C 用户态组合，具有显著优势：

    对比维度                      libbpf + C/Go                        Aya (Rust)

语言统一性                    内核态 C，用户态                    内核态与用户态同为 Rust，类型自动共享
依赖体积                     Go/C，类型需手动同步                 纯 Rust 静态链接 3-5 MB，无 CGO
内存安全                                                  编译期保证内存安全，无数据竞争
                         Go + libbpf 约 8-15 MB，需 CGO

                         C 代码易出 UAF、缓冲区溢出

生态成熟度                    极成熟，Cilium 等大型项目验证           快速成长中，已可用于生产

                              表 2-3 libbpf 与 Aya 框架对比

选择                       Aya                          的核心理由：单语言栈带来的开发效率提升（约

30%）和类型安全保证，在长期维护和社区贡献方面优势明显。代价是 Aya 生态不如 libbpf

eBPF 验证器在不同内核版本间的行为差异，这通过 CI 测试矩阵可以覆盖。

3.1.1 真实且高频的用户痛点

3.1.2 10-50 倍的性能代差优势

XDP 与 iptables 的性能差距不是渐进式的，而是代际的。在 100 万 PPS SYN Flood 场景下：

• iptables 方案：CPU 占用 80-100%，系统接近不可用，正常请求超时。

• fail2ban + iptables：因日志分析延迟，前 5-10 秒攻击完全穿透；之后 CPU 仍 60%+。

• eShield (XDP)：CPU 占用 5-10%，正常请求零影响，攻击包在驱动层即被丢弃。

这种数量级的差距意味着                                                  eShield
不是"略好一点"的替代品，而是"能用与不能用"的差别。对于已经被                                   CC
打到服务不可用的用户，eShield 是救命稻草级别的工具。

3.1.3 极高的开源传播潜力

3.2 反对开发的论据与风险

3.2.1 云黑洞机制：无法逾越的物理天花板

这是最核心的反对理由。主流云厂商（阿里云、腾讯云、AWS）都有"黑洞机制"：当检测到某台主

机的入流量超过阈值（阿里云基础防护 2 Gbps，腾讯云 2-10 Gbps，AWS Shield Standard

不限速但有其他限制），为了保护机房其他租户，云厂商会在上层路由直接将该                       IP

的所有入流量丢弃，持续 30 分钟到 24 小时不等。

这意味着：如果攻击者打出 10 Gbps 的 UDP Flood，流量根本到不了你的 eShield 程序——云厂
商在更上游就把你"拔网线"了。你的程序性能再高，连表现的机会都没有。因此，eShield
绝对不能宣传为"DDoS 防御系统"，否则会引发严重的用户预期错位。

关键认知：主机级防御对抗不了"带宽耗尽型"攻击。这是物理定律，不是工程问题。eShield
的战场必须是"带宽没满但 CPU/连接数被耗尽"的 CC 攻击场景。

3.2.2 eBPF 验证器的跨版本兼容性负担

eBPF                     验证器（verifier）在不同内核版本间存在行为差异。一段在  5.15

内核上通过验证的代码，可能在 5.10 上因为循环展开策略不同而被拒绝；一段在 6.1

上利用了新特性的代码，在 5.4 上根本无法加载。这意味着：

• 需要维护一个 5.10 / 5.15 / 6.1 / 6.6 的内核版本测试矩阵

• CI 需要用 QEMU 或 Vagrant 跑多版本回归测试，CI 时间显著增加
• 部分高级特性（如 bpf_loop 助手、bpf_for_each_map_elem）需要做版本探测和降级
• 用户反馈"在我的内核上跑不起来"的 issue 会持续消耗精力

3.2.3 安全风险：内核态代码的责任

eBPF 程序运行在内核态，虽然验证器保证了"不会直接 crash 内核"，但逻辑错误仍可能导致：

• 误杀合法流量：速率限制阈值设置不当，封禁正常用户，引发投诉

• 性能反噬：XDP 程序本身写得低效（如频繁 Map 查找），反而拖累所有流量

• 被滥用：工具被攻击者用于构造反射攻击（虽然 XDP 主要做入站过滤，但 XDP_TX
可构造响应包）

应对措施：默认"观察模式"运行                                           24

小时再切换防护；丰富的白名单语法；开源协议加入反滥用条款；代码审计与社区监督。

3.2.4 长期维护成本

开源项目的维护成本常被低估。eBPF 项目尤其如此：内核每个 LTS 版本都会引入新特性、废弃旧
助手函数、调整验证器行为。维护者需要持续跟进内核演进，否则项目会在 2-3 年内因兼容性问题
而"腐烂"。如果维护者精力不足，项目可能重蹈许多"明星项目"半年无更新的覆辙。

   决策结论：建议开发，但必须精准定位。

// eBPF 内核态入口（Rust + Aya 风格伪代码）
#![no_std]
#![no_main]

use aya_ebpf::{bindings::xdp_action, programs::XdpContext};
use aya_log_ebpf::info;
use core::mem;

const ETH_HDR_LEN: usize = 14;
const IP_HDR_LEN: usize = 20;
const TCP_HDR_LEN: usize = 20;

#[repr(C)]
struct EthHdr {
dst: [u8; 6],
src: [u8; 6],
proto: u16,
}

#[repr(C)]
struct Iphdr {
ver_ihl: u8,
tos: u8,
tot_len: u16,
id: u16,
frag_off: u16,
ttl: u8,
proto: u8,
check: u16,
saddr: u32,
daddr: u32,
}

// 安全的指针读取：带边界检查
unsafe fn ptr_at<T>(ctx: &XdpContext, off: usize) -> Option<*const T> {
let start = ctx.data();
let end = ctx.data_end();
if start + off + mem::size_of::<T>() > end {
return None;
}
Some((start + off) as *const T)
}

 #[xdp]
 pub fn eshield(ctx: XdpContext) -> u32 {
 match try_eshield(&ctx) {
 Ok(act) => act,
 Err(_) => xdp_action::XDP_PASS, // 失败默认放行，避免误杀
 }
 }

 fn try_eshield(ctx: &XdpContext) -> Result<u32, ()> {
 // 1. 解析以太网头
 let eth: *const EthHdr = unsafe { ptr_at(ctx, 0) }.ok_or(())?;
 if unsafe { (*eth).proto } != 0x0008 { // htons(ETH_P_IP) = 0x0800
 return Ok(xdp_action::XDP_PASS); // 非 IPv4，放行
 }

 // 2. 解析 IP 头
 let ip: *const Iphdr = unsafe { ptr_at(ctx, ETH_HDR_LEN) }.ok_or(())?;
 let ip_hdr_len = (unsafe { (*ip).ver_ihl } & 0x0f) as usize * 4;
 if ip_hdr_len < IP_HDR_LEN {
 return Ok(xdp_action::XDP_DROP); // 非法 IP 头长度
 }

 let saddr = unsafe { (*ip).saddr };
 let proto = unsafe { (*ip).proto };

 // 3. 白名单检查（快速路径）
 if is_whitelisted(saddr) {
 return Ok(xdp_action::XDP_PASS);
 }

 // 4. 黑名单检查
 if is_blacklisted(saddr) {
 emit_event(saddr, EVENT_BLACKLIST_HIT);
 return Ok(xdp_action::XDP_DROP);
 }

 // 5. 速率限制
 if exceeds_rate_limit(saddr) {
 add_to_blacklist(saddr, BLOCK_DURATION_SHORT);
 emit_event(saddr, EVENT_RATE_LIMIT_HIT);
 return Ok(xdp_action::XDP_DROP);
 }

 // 6. 协议级处理
 match proto {
 6 => handle_tcp(ctx, ip, ip_hdr_len), // TCP
 17 => handle_udp(ctx, ip, ip_hdr_len), // UDP
 _ => Ok(xdp_action::XDP_PASS),
 }
 }

     • 事件上报最小化：仅在 DROP 时上报事件，PASS 路径零事件开销。

4.2.2 速率限制：Per-IP 滑动窗口

速率限制是 CC 防御的核心。eShield 采用 Per-CPU LRU Hash +
滑动窗口计数器的组合方案，兼顾精度与性能。

算法选择：滑动窗口（Sliding Window）比固定窗口（Fixed Window）更精确，避免了窗口边
界处的突发流量问题。但传统滑动窗口需要维护时间戳列表，内存开销大。eShield
采用指数衰减滑动窗口（Exponential
Decay），仅需存储一个计数器和一个时间戳，内存开销恒定，且天然支持 Per-CPU 聚合。

 // 速率限制：指数衰减滑动窗口
 // 原理：counter = counter * decay_factor + 1，每 TICK_MS 毫秒衰减一次
 // decay_factor = 0.5 表示"半衰期 = TICK_MS"，即 1 秒前的计数权重减半

 const TICK_MS: u64 = 100; // 衰减周期 100ms
 const DECAY_NUM: u64 = 7; // 衰减分子（7/8 = 0.875）
 const DECAY_DEN: u64 = 8; // 衰减分母
 const RATE_THRESHOLD: u64 = 200; // 单 IP 每 TICK 窗口阈值

 #[repr(C)]
 struct RateCounter {
 counter: u64, // 衰减后的计数
 last_decay_ns: u64, // 上次衰减时间（纳秒）
 padding: [u8; 8], // 缓存行对齐填充
 }

 // Per-CPU LRU Hash：key = src_ip, value = RateCounter
 // 使用 LRU 避免内存无限增长，max_entries = 100000
 #[map]
 static RATE_MAP: LruHashMap<u32, RateCounter> = LruHashMap::pinned(100000, 0)?;

 fn exceeds_rate_limit(saddr: u32) -> bool {
 let now = unsafe { bpf_ktime_get_ns() };
 let now_ms = now / 1_000_000;

 let entry = match RATE_MAP.get(&saddr) {
 Some(e) => e,
 None => {
 // 新 IP，初始化计数器
 let counter = RateCounter {
 counter: 1,
 last_decay_ns: now,
 padding: [0; 8],
 };
 let _ = RATE_MAP.insert(&saddr, &counter, 0);
 return false; // 首个包不触发

}
};

// 计算需要衰减的 tick 数
let elapsed_ms = now_ms.saturating_sub(entry.last_decay_ns / 1_000_000);
let ticks = elapsed_ms / TICK_MS;

// 应用衰减（最多 64 个 tick，避免极端情况下的长循环）

let mut decayed_counter = entry.counter;
for _ in 0..effective_ticks {
decayed_counter = (decayed_counter * DECAY_NUM) / DECAY_DEN;
}

// 增加当前包计数
decayed_counter = decayed_counter.saturating_add(1);

// 更新 Map

counter: decayed_counter,
last_decay_ns: now,
padding: [0; 8],
};
let _ = RATE_MAP.insert(&saddr, &updated, 0);

// 判定
decayed_counter > RATE_THRESHOLD
}

4.2.3 SYN Proxy：Cookie 挑战机制

SYN Flood 是最常见的 L4 攻击。攻击者发送海量伪造源 IP 的 SYN
包，服务器为每个包分配半连接（TCB），很快耗尽连接表。eShield 的 SYN Proxy 在 XDP
层实现 Cookie 挑战，让伪造源 IP 的 SYN 包根本无法消耗服务器资源。

核心思路：对到达的 SYN 包，不立即交给内核协议栈，而是用 SYN-ACK Cookie 回复。Cookie
编码了连接关键信息（源
IP、源端口、时间戳、MSS），用服务器密钥签名。只有真正想建立连接的客户端才会回复
ACK（携带 Cookie），攻击者伪造源 IP 收不到 SYN-ACK，自然无法完成挑战。

// 验证 ACK 包携带的 Cookie
fn handle_ack_with_cookie(ctx: &XdpContext, ip: *const Iphdr, ip_hdr_len: usize) -> Result<u32, ()> {

// 验证当前和上一个 bucket

let daddr = unsafe { (*ip).daddr };
let sport = u16::from_be(unsafe { (*tcp).source });
let dport = u16::from_be(unsafe { (*tcp).dest });

let current_bucket = now_s / BUCKET_DURATION_S;

for offset in 0..VALID_BUCKETS {
let bucket = current_bucket.saturating_sub(offset as u64);
let secret_bytes = if bucket == secret.bucket_index {
&secret.current
} else if bucket == secret.bucket_index.saturating_sub(1) {
&secret.previous
} else {
continue;
};

let mss = (expected >> 24) as u16; // MSS 编码在 Cookie 高 8 位
let computed = syncookie_hash(saddr, daddr, sport, dport, bucket, mss, secret_bytes);

if computed == expected {
// Cookie 验证通过，放行让协议栈建立连接
// 可选：重写 seq 号，让协议栈无感知
return Ok(xdp_action::XDP_PASS);
}
}

// Cookie 无效，可能是攻击或过期，丢弃
Ok(xdp_action::XDP_DROP)

4.2.4 L7 轻量指纹扫描

L7 扫描是 eShield 的可选高级特性。XDP 层做完整 L7 检测不现实（验证器限制 +
性能开销），但可以做"轻量指纹"——读取 TCP 载荷前 64 字节，匹配固定模式（如异常
User-Agent 前缀、已知攻击工具特征）。命中则丢弃或重定向到用户态 L7 代理做深度检测。

 // L7 轻量指纹扫描（仅读取前 64 字节）
 const L7_SCAN_BYTES: usize = 64;

 #[repr(C)]
 struct L7Pattern {

return false;
}
}

4.3.4 自适应阈值引擎

固定阈值难以适应不同站点的流量特征。自适应引擎维护过去                                                   24

小时的流量基线（按小时分桶的均值和标准差），当实际流量超过基线                                               N

倍时自动提升阈值或告警。初期保守，宁可漏过少量攻击，不可大规模误杀。

// 自适应阈值引擎
use std::collections::VecDeque;

const HISTORY_HOURS: usize = 24;
const BASELINE_MULTIPLIER: f64 = 3.0; // 基线的 3 倍触发告警
const ADAPTIVE_MULTIPLIER: f64 = 5.0; // 基线的 5 倍自动提升阈值

struct AdaptiveEngine {
// 按小时分桶的历史 PPS（每桶 1 小时，保留 24 桶）
hourly_pps: VecDeque<f64>,
current_hour_pps: f64,
current_hour_start: chrono::DateTime<chrono::Utc>,
}

impl AdaptiveEngine {
fn record(&mut self, current_pps: f64, now: chrono::DateTime<chrono::Utc>) {
// 检查是否跨小时
if now.hour() != self.current_hour_start.hour() {
self.hourly_pps.push_back(self.current_hour_pps);
if self.hourly_pps.len() > HISTORY_HOURS {
self.hourly_pps.pop_front();
}
self.current_hour_pps = 0.0;
self.current_hour_start = now;
}
self.current_hour_pps = self.current_hour_pps.max(current_pps);
}

fn compute_baseline(&self) -> (f64, f64) { // (mean, std_dev)
if self.hourly_pps.len() < 6 {
return (0.0, f64::MAX); // 数据不足，不触发
}
let n = self.hourly_pps.len() as f64;
let mean = self.hourly_pps.iter().sum::<f64>() / n;
let variance = self.hourly_pps.iter()
.map(|x| (x - mean).powi(2))
.sum::<f64>() / n;
(mean, variance.sqrt())
}

fn evaluate(&self, current_pps: f64) -> Action {
let (mean, std) = self.compute_baseline();
if mean == 0.0 { return Action::None; }

let z_score = (current_pps - mean) / std.max(1.0);

match z_score {
z if z > ADAPTIVE_MULTIPLIER => Action::RaiseThreshold(current_pps as u64),
z if z > BASELINE_MULTIPLIER => Action::Alert,
_ => Action::None,
}
}
}

enum Action {
None,
Alert,
RaiseThreshold(u64),
}

4.4.2 关键 Map 设计详解

WHITELIST (LPM Trie)
LPM Trie（Longest Prefix Match）是 CIDR 匹配的天然数据结构。查询复杂度 O(地址位数)，即
IPv4 的 32 次比较，与规则数量无关。相比 Hash Map 只能精确匹配，LPM Trie
支持"10.0.0.0/8"这样的网段白名单。容量 1024 条对绝大多数场景足够。

BLACKLIST (LRU Hash)
使用 LRU（Least Recently Used）而非普通 Hash，是为了防止 Map 无限增长。当达到 100,000
容量时，最久未访问的 IP 会被自动驱逐。BlockEntry 结构包含：

#[repr(C)]
struct BlockEntry {
blocked_until_ns: u64, // 封禁到期时间（0 = 永久）
block_reason: u8, // 0=手动, 1=速率限制, 2=SYN Flood, 3=L7
hit_count: u32, // 命中次数（用于展示）
first_seen_ns: u64, // 首次封禁时间
_pad: [u8; 3],
}

RATE_LIMIT (Per-CPU LRU Hash)
Per-CPU Map 是性能关键。每个 CPU 有独立的 Map 实例，写入无锁，避免了多核间的 cache
line 争用。代价是用户态聚合时需要遍历所有 CPU 的 Map 求和。在 32 核机器上，聚合 100,000
个 IP 的统计约需 50ms，完全可接受（每 100ms 聚合一次）。

EVENTS (Ring Buffer)

Ring Buffer 大小 4 MB，可缓存约 120,000 个事件（每个 32 字节）。在 100 万 PPS

攻击下，即使每包都产生事件，也能缓存约 0.12 秒的数据。用户态消费速度（约 200

万事件/秒）远超生产速度，不会反压。如果用户态确实消费不过来，Ring                                    Buffer

会丢弃旧事件并记录 dropped_events 计数。

4.5 性能优化策略

eShield 的性能目标是：2C4G 云主机在 100 万 PPS SYN Flood 下 CPU 占用 <
10%。达成这一目标需要在多个层面进行优化。

4.5.1 缓存行对齐

x86 CPU 的缓存行为 64 字节。如果两个频繁访问的字段落在同一缓存行，多核间的写入会导致
cache line bouncing，性能下降 10-100 倍。所有 Per-CPU Map 的 Value
结构都必须显式对齐到缓存行边界。

 // 缓存行对齐的统计结构
 use core::mem::size_of;

 const CACHE_LINE: usize = 64;

 rate_limited: u64, // 8

4.5.2 Per-CPU 统计聚合

全局统计使用 Per-CPU Array，每个 CPU 独立累加，无锁。用户态聚合时一次性读取所有 CPU
的值求和。在 32 核机器上，读取 32 个 64 字节的缓存行仅需约 2 微秒，远低于 100ms
的聚合周期。

4.5.3 批量事件处理

Ring Buffer 事件消费采用批量模式：单次系统调用读取最多 256 个事件，减少 user/kernel
切换开销。日志写入也批量进行，每 256 个事件或 100ms 触发一次 flush。

4.5.4 快速路径与慢速路径分离

XDP 程序的代码路径分为快速路径（Fast Path）和慢速路径（Slow Path）：

• 快速路径：白名单命中、正常放行的包。代码尽量短，仅必要的 Map 查询。

• 慢速路径：触发速率限制、SYN Proxy、L7 扫描的包。可以执行更复杂的逻辑。

•  分支预测友好：使用             likely/unlikely  提示（Rust  中通过  #[cold]

属性），让编译器优化常见路径。

5.3 法律合规风险

安全工具天然存在"双刃剑"属性。eShield 虽然是防御工具，但仍需注意：
     • 开源协议：采用 Apache 2.0 + 补充反滥用声明，明确禁止用于攻击目的
     • 免责条款：README 和 LICENSE 中声明"不保证 100% 防御，使用者自担风险"
     • 数据隐私：事件日志仅记录源 IP 和协议信息，不记录应用层载荷（L7 扫描仅匹配不存储）
     •
     地区合规：文档注明"使用者需遵守所在地区网络安全法规"，不提供绕过任何安全机制的功能
